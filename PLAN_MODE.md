# PLAN_MODE.md

## 이 문서의 용도
Claude Code의 **플랜모드 세션** 에서 이 프롬프트를 복사해 사용한다.
본격 구현을 시작하기 전, 설계의 빈틈과 리스크를 검증하기 위한 용도다.
한 번에 모든 것을 묻지 않고 **3단계** 로 나누어 진행한다.

---

## 사전 조건 (읽어야 할 문서)
Claude는 플랜모드 진입 시 다음을 **반드시** 먼저 읽는다.
- `CLAUDE.md`
- `docs/ARCHITECTURE.md`
- `docs/DATA_MODEL.md`
- `docs/DOMAIN_RULES.md`
- `docs/SECURITY.md`

---

## 공통 지시사항 (모든 단계에 적용)
1. **가정·질문을 먼저 출력하라.** 모호한 부분을 추측으로 메우지 말고 사용자에게 역질문.
2. **주요 선택지는 장단점·추천·근거를 ADR 형식** 으로 제시.
3. **결정의 되돌림 난이도** 를 명시: `[reversible]` / `[hard-to-reverse]` / `[irreversible]`
4. **비개발자가 현장에서 범할 수 있는 실수** 를 예측하고 설계로 방지.
5. **토큰·연산 비용 과잉** 을 경계. 간결한 구현 우선.
6. **불확실성은 "가정"이라고 명시** 하고 확인 요청.
7. 표보다 **계층형 목록** 으로 출력.
8. 모든 응답은 **한국어**.

---

## Stage 1: 전략·설계 확정

### 질문
아래 항목을 순서대로 출력하라. 이미 `CLAUDE.md`와 `docs/*.md`에 결정된 사항은 재확인 수준으로 짧게 언급하고, 문서에 없는 사항 위주로 심층 설계한다.

1. **요구사항 재정의** — 현재 문서 기준 요구사항을 1페이지로 요약. 누락·모호한 부분 지적.
2. **MVP 범위 / Future 범위** — 각 기능을 둘 중 하나로 분류. 경계선상 항목은 이유 설명.
3. **사용자 역할 정의** — 관리자, 참석자 외 추가 역할 필요성 검토 (예: 발표자 전용 모드).
4. **핵심 사용자 시나리오** — 5가지 이상. 성공 경로 + 실패·예외 경로 각 1개씩.
5. **아키텍처 개요** — `ARCHITECTURE.md` 기반으로 데이터 흐름과 의존성 도식(텍스트 트리).
6. **리스크 Top 10** — 기술·운영·법무·UX 각 영역 포함. 각 리스크에 완화책 1줄.
7. **열린 질문 목록** — 사용자 결정이 필요한 미정 사항.

### 이 단계 종료 조건
사용자가 "Stage 1 확정"이라고 답할 때까지 수정·재작성 반복.

---

## Stage 2: 상세 설계

### 질문
Stage 1이 확정된 후 진행. 다음을 **구현 가능한 수준** 으로 작성.

1. **데이터 모델 상세** — `DATA_MODEL.md` 를 기반으로 각 엔티티의 전체 필드·타입·제약 확정.
2. **API 설계 상세** — 엔드포인트별 요청·응답 스키마(Zod 수준), 에러 코드 정의.
3. **화면 목록과 컴포넌트 트리** — 참석자 화면 5~7개, 관리자 화면 5~7개 예상.
4. **권한 모델** — RLS 정책, 관리자 API의 인증 미들웨어 설계.
5. **상태 머신** — Event 상태(`draft → published → checkin_open → checkin_closed → survey_open → archived`)와 각 상태에서 허용되는 행위.
6. **에러 처리 정책** — 사용자 노출 메시지 vs 내부 로그 분리, 실패 체크인의 관찰 가능성.
7. **테스트 전략** — 필수 유닛·E2E 시나리오 목록.

### 이 단계 종료 조건
사용자가 "Stage 2 확정"이라고 답할 때까지 수정·재작성 반복.

---

## Stage 3: 실행 계획

### 질문
Stage 2 확정 후 진행.

1. **개발 단계별 일정** — 주 단위 마일스톤 (M1: 스키마·체크인 / M2: Q&A·설문 / M3: 관리자 대시보드 / M4: 배포·테스트).
2. **첫 구현에서 생성할 파일·폴더 구조** — 전체 트리와 각 파일의 목적 1줄.
3. **환경 변수 목록** — `.env.example` 내용.
4. **배포 체크리스트** — Vercel·Supabase 초기 설정 순서.
5. **첫 행사 운영 리허설 체크리스트** — 실제 행사 전 테스트 시나리오.
6. **RUNBOOK.md 초안** — 현장 장애 대응(체크인 실패, Wi-Fi 장애, 관리자 로그인 불가) 시 조치 순서.

### 이 단계 종료 조건
사용자가 "Stage 3 확정"이라고 답하면 실제 코드 구현 단계로 전환.

---

## Stage별 체크리스트 (Claude 자체 검증용)

### Stage 1 자기점검
- [ ] 모든 가정을 "가정:" 으로 명시했는가?
- [ ] 리스크에 완화책을 붙였는가?
- [ ] MVP와 Future 경계선상 항목의 결정 근거를 제시했는가?

### Stage 2 자기점검
- [ ] 데이터 모델에 `created_at`, `retention_until` 필드를 포함했는가?
- [ ] 휴대폰 번호 필드에 암호화 표기를 넣었는가?
- [ ] 익명 Q&A 응답 DTO에 세션 토큰이 **빠져** 있는가?
- [ ] 관리자 API와 참석자 API를 분리했는가?
- [ ] 상태 머신의 모든 전이에 허용 조건을 명시했는가?

### Stage 3 자기점검
- [ ] 모든 환경 변수를 `.env.example`에 반영했는가?
- [ ] 첫 구현 파일 트리가 `ARCHITECTURE.md` 폴더 규칙을 따르는가?
- [ ] RUNBOOK에 비개발자도 수행 가능한 수준의 절차가 적혔는가?

---

## 출력 스타일
- 한국어
- 표보다 계층형 목록
- 불필요한 장문·수식어 지양
- 코드 예시는 꼭 필요한 곳에만, 최소한으로
- 결정사항에는 근거(ADR) 1~2줄 첨부

---

## 사용 예시
사용자 첫 메시지:
> "PLAN_MODE.md와 CLAUDE.md, docs/*.md를 읽고 Stage 1을 시작해줘."

이후 Stage 1 → Stage 2 → Stage 3 순서로 진행한다. 각 Stage가 확정되기 전까지 다음 단계로 넘어가지 않는다.
